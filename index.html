Core Strategy

Parsing Mechanics

We implemented a top-down **Recursive Descent Parser**. In this approach, every rule of our context-free grammar maps to a specific helper method in our Parser.java implementation. It acts as an $LL(k)$ architecture, utilizing localized lookahead arrays to make deterministic decisions before executing syntactic evaluations.

Language Used: Java 17+ (Swing GUI)
Parsing Category: Top-Down, Backtrack-Free
Exception Type: Runtime SyntaxException
Avoiding Infinite Recursion

Left Recursion Resolution

Left recursion causes top-down parsers to enter infinite lookup loops. We bypassed this classic design pitfall using two primary methods:

  • Grammar Refactoring: Direct translation of rules to eliminate leading recursions on statements.
  • Precedence Climbing: Arithmetic evaluation functions parse operations iteratively inside sequential loops rather than looping back into identical function methods.

Token Collection

Tokens are stored sequentially inside a java.util.ArrayList<Token> structure, providing $O(1)$ lookup performance for lookahead operations inside the parser.

Comments Handling

Single-line comments (//) are processed directly inside the Lexer's scanning method. The pointer discards all characters until a carriage return or newline (\n) is reached, generating zero tokens.

IDs vs. Keywords

The scanner implements the **Longest Match** pattern. Once an alphanumeric sequence is fully constructed, it is evaluated against a fixed HashSet of reserved words. If flagged, it is classified as a Keyword; otherwise, it is generated as an Identifier.

Advanced Edge-Case Scenarios

Comment Only Inputs

If only comments or whitespace are entered, the Lexer completes with an empty collection. The Parser immediately identifies an EOF token and constructs an empty structure.

Alphanumeric Identifiers

Inputs like abc123 are evaluated successfully. Because the leading element is a valid letter, the entire alphanumeric string is parsed as one coherent identifier token.

Input Limits (>15 Characters)

To limit code inputs (e.g., rejecting files above a certain character size), we intercept inputs directly inside the GUI Action Listener, blocking analysis before parsing begins.

Value Nature

Data Type

Defines the storage capacity, internal format, and mathematical operations that can be executed directly on the identifier.

Supported Types: int, float, void.

Primary Role: Governs mathematical operations and guides memory layout allocations.

Source Context: Directly specified in explicit code declaration statements (e.g. float y = 5.5;).

Structural Role

Symbol Type

Defines the functional structural architecture of the identifier entity inside the overall program construct.

Supported Categories: VARIABLE, FUNCTION.

Primary Role: Tells the Parser if an identifier is a storage address or executable code block.

Source Context: Inferred automatically by the Parser based on lookahead token rules.

Common Trap Questions

"Why is your Symbol Table a flat HashMap instead of supporting nested scopes?"

Strategy: "Our project scope focuses strictly on **Frontend Analysis** (Lexer and Parser). A flat, highly performant $O(1)$ lookup table allows us to demonstrate type and identifier mapping clearly, while delegating nested scope structures to a downstream Semantic phase."

"Why does the compiler accept declarations like 'void x = 10;' without warning?"

Strategy: "This highlights the boundary between **Syntax** and **Semantics**. Syntactically, void x = 10; matches the structural grammar rule Type Identifier Assign Literal Semicolon. Since our compiler focuses on syntax validation, it approves the structure. Type consistency checking is a Semantic phase task."

"Why does declaring a duplicate variable not throw a compilation error?"

Strategy: "Duplicate identifier detection is a semantic validation rule. Syntactically, repeating a valid declaration statement is perfectly grammatical. Our parser builds the correct parse tree for both statements, while the underlying Symbol Table registers the declarations."
